### 1. 核心设计理念："混合架构"与"便携解耦"
*   **模型与脚本分工**：摒弃纯脚本正则提取或纯大模型处理。Python 脚本负责物理操作（解压、备份、查表、回写），大模型负责语义理解（判断代码中哪些是玩家可见文本、结合语境翻译）。
*   **插件与目标解耦**：插件本体（脚本、CSV 术语表）常驻插件目录，可对 VS Code 中打开的任意整合包目录执行翻译，所有生成的临时文件、备份和翻译结果均存放在目标整合包目录下。

### 2. 兼容性与提取策略
*   **全量支持**：不能只盯 JAR 和 FTB Quests。必须兼容各版本模组语言文件（1.12 的 `.lang` 与 1.16+ 的 `.json`）。
*   **脚本与代码安全**：支持低版本的 CraftTweaker (`.zs`) 和高版本的 KubeJS (`.js`)，以及配置文件 (`.toml`/`.cfg`)。大模型需精准提取玩家可见字符串，**绝对不能破坏代码语法结构**。JSONL 中通过 `context_hint` 字段记录锚点信息，供回写脚本精准定位替换位置。
*   **编码统一**：所有文件读写统一使用 **UTF-8** 编码，脚本和大模型读写均遵循此规则。

### 3. 资源应用与兜底机制（零侵入）
*   **绝不修改原版 JAR**：从模组中提取的翻译不塞回原 JAR 包。
*   **动态探测输出**：脚本同时检测 `kubejs/` 目录和 `mods/kubejs-*.jar`，任一条件满足即判定为 KubeJS 模式。
    *   **优先方案**：若有 KubeJS，将翻译生成到 `kubejs/assets/` 目录，游戏自动加载。
    *   **兜底方案**：若无 KubeJS，自动生成标准的 `resourcepacks/Auto_Translation.zip` 资源包，提示用户手动加载。
*   **已有翻译跳过**：若目标模组已存在 `zh_cn.json` 或 `zh_CN.lang`，跳过该模组不覆盖，避免陈旧翻译干扰。

### 4. 灾备与安全回滚（双保险）
*   **全量备份**：在执行任何修改前，脚本必须将目标目录中所有将被修改的文件（包括被合并覆盖的已存在翻译文件）备份至 `_lang_backup/`，并生成 `backup_manifest.json`。清单中需单独记录已存在的 KubeJS 翻译文件，以便回滚时区分新旧文件。
*   **一键回滚**：提供独立的 `rollback.py` 脚本，对照清单还原备份文件，仅清理翻译过程中新生成的文件（保留翻译前就存在的 KubeJS 翻译文件），并清除 `Auto_Translation.zip`、`_temp_extracted/` 等临时资源。

### 5. 术语表与 Token 优化
*   **大容量本地查表**：10 万级别的术语表保留为 **CSV 格式**（便于人工维护），以 **EN→ZH** 映射为主，**不进入大模型上下文**。
*   **内存索引与多义词**：由 Python 脚本通过内存索引（Pickle 缓存）极速查询，支持返回一词多义 JSON 数组，大模型根据语境选择最合适的释义。

### 6. 中间体数据格式
*   **弃用 CSV，改用 JSONL**：大模型提取的翻译任务清单和翻译结果，必须使用 **JSONL (JSON Lines)** 格式作为中间载体。彻底解决 MC 文本中常见的换行符、双引号、逗号导致 CSV 结构炸裂的问题，提升鲁棒性。
*   **JSONL 字段定义**：`{"id": 1, "source_file": "相对路径", "key": "键名", "original": "原文", "translated": "", "status": "pending", "context_hint": "上下文提示"}`
*   **更新方式**：翻译完成后根据 ID 原地更新对应行，避免追加导致重复记录。
*   **分批策略**：初期每批最少处理 200 行，后期经测试确认稳定后升级为全量处理。
