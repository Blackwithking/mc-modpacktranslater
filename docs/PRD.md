# 文档一：Minecraft 整合包智能本地化插件 - 需求文档 (PRD) v2.1
## 1. 项目背景与目标
Minecraft 整合包的汉化面临文件格式繁杂（JAR、JSON、SNBT、JS、ZS等）、版本差异大、任务量庞大且术语不统一等痛点。传统纯脚本正则提取极易破坏代码逻辑，纯大模型处理又面临 Token 消耗高与文件读写崩溃的问题。术语表以 EN→ZH 映射为主。
**本项目旨在开发一款 Codex 插件，采用"混合架构"（脚本负责物理操作，大模型负责语义理解），实现整合包全量文本的安全、智能、低成本翻译。插件需做到即插即用，可指向任意游戏目录执行操作。**
## 2. 核心设计理念
- **便携解耦**：插件脚本与 10 万级术语表常驻插件目录，可对 VS Code 中打开的任意整合包目录执行翻译，不污染插件本体。
- **模型干智活，脚本干脏活**：大模型负责识别代码中玩家可见的文本并翻译；脚本负责解压、备份、术语查表及安全回写。
- **零侵入与动态兜底**：绝不修改原始模组 `.jar` 文件。自动探测环境，有 KubeJS 则生成外部覆盖资源，无则生成标准资源包 ZIP 兜底。
- **安全中间体**：采用 JSONL (JSON Lines) 作为翻译任务中间载体，彻底解决 CSV 在处理换行符、引号和颜色代码时易发生的结构炸裂问题。
- **全量备份与一键回滚**：在目标目录生成备份清单，翻译翻车可一键恢复至初始状态。
## 3. 功能需求
### 3.1 物理准备与备份模块 (`prepare.py`)
- **参数化执行**：接收目标整合包路径作为参数。
- **目录扫描**：遍历目标目录，生成 `file_tree.json`。
- **JAR 只读解包**：提取 `.jar` 中的 `en_us.json` / `en_US.lang` 到目标目录的 `_temp_extracted/` 供模型读取。所有文件读写统一使用 **UTF-8** 编码。
- **全量安全备份**：将目标目录中所有将被修改的文件（`.json`, `.lang`, `.snbt`, `.zs`, `.js`, `.cfg`, `.toml` 及已存在的 KubeJS 翻译文件）备份至目标目录的 `_lang_backup/`，并生成 `backup_manifest.json`（记录所有备份文件的原始路径及已存在的 KubeJS 翻译文件清单，供回滚时区分新旧文件）。
### 3.2 智能提取模块 (大模型执行)
- 读取 `file_tree.json` 及文件内容，根据文件类型差异化提取：
  - **模组语言文件**：提取 Value。
  - **KubeJS / CraftTweaker**：仅提取玩家可见字符串，绝对排除变量名、函数名。
  - **FTB Quests / 配置文件**：提取 title/description 或注释说明。
- **输出为 `translation_tasks.jsonl`**（每行一个 JSON 对象，格式：`{"id": 1, "source_file": "相对路径", "key": "键名", "original": "原文", "translated": "", "status": "pending", "context_hint": "上下文提示"}`）。
  - `source_file`：源文件相对于整合包根目录的路径。
  - `key`：语言键名（语言文件中）或代码中该文本的定位标识。
  - `context_hint`：用于代码文件替换时的安全锚点（如行号、前后文特征），帮助 generate_assets.py 精准定位替换位置，避免误伤变量名/函数名。
### 3.3 术语查询模块 (`glossary_lookup.py`)
- **输入**：接收 `translation_tasks.jsonl` 文件路径和 ID 范围（如 `--ids 1-200`），脚本自行读取 JSONL 中对应行的 `original` 字段，拼接后查询术语表。
- **零 Token 消耗**：10 万行 CSV 不进入大模型上下文，由脚本进行内存索引（Pickle 缓存生成在插件目录）。
- **多义词支持**：按 ID 逐条返回对应的术语结果，每条包含一词多义 JSON 数组。
### 3.4 语境翻译模块 (大模型执行)
- **分批处理**：每批读取最少 200 行 JSONL 记录，调用查表脚本获取术语。后期经测试确认稳定后可升级为全量处理。
- **多义词选择**：大模型根据完整语境选择最合适的释义。
- **格式保留**：严格保留 `%s`, `%1$s`, `§a`, `&b` 等占位符和颜色代码。
- **安全更新**：将翻译结果按 ID 原地更新至 `translation_tasks.jsonl` 对应行的 `translated` 字段，`status` 改为 `done`。
### 3.5 动态回写与生成模块 (`generate_assets.py`)
- **环境探测**：同时检测目标目录下是否存在 `kubejs/` 目录以及 `mods/` 中是否存在 `kubejs-*.jar`，任一条件满足即判定为 KubeJS 模式。
- **已有翻译处理**：若目标模组已存在 `zh_cn.json` 或 `zh_CN.lang`，跳过不处理（避免陈旧翻译覆盖）。
- **模组语言文件输出**：
  - *KubeJS 模式*：合并写入 `kubejs/assets/<modid>/lang/zh_cn.json`。
  - *兜底模式*：构建含 `pack.mcmeta` 的标准结构，打包为 `resourcepacks/Auto_Translation.zip`。
- **代码/配置文件替换**：对 `.js`, `.zs`, `.snbt` 利用 `context_hint` 字段中的锚点信息执行**绝对字符串匹配替换**，不改动任何代码逻辑符号。
### 3.6 灾备回滚模块 (`rollback.py`)
- 读取目标目录的 `backup_manifest.json`。
- 将 `_lang_backup/` 中的文件还原至原始路径。
- 清理脚本新生成的文件：删除 `Auto_Translation.zip`、不在备份清单中的 KubeJS 翻译文件（即翻译过程中新生成的）、`_temp_extracted/` 等临时目录。
